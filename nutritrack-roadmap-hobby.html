Roadmap hobbistica
per Nutritrack

Piano di sviluppo revisionato per uso personale e condivisione con conoscenti. Niente backend, niente account, niente monetizzazione, niente store review. Solo feature che migliorano l'esperienza quotidiana per te e chi ti sta vicino. Tutte le implementazioni restano nel perimetro PWA + localStorage.

🏠

Local-first forever

Nessun backend. I dati restano su localStorage. Sync solo via export/import JSON manuale o QR code tra dispositivi propri.

🆓

Zero costi ricorrenti

Nessun server da pagare, nessuna API key a consumo. Solo GitHub Pages gratuito e API Open Food Facts (gratis, no key).

👤

Single contributor

Effort realistico per uno sviluppatore nel tempo libero. Nessuna dipendenza da terzi per manutenzione.

🎯

Feature user-driven

Priorità alle funzionalità che usi davvero o che i tuoi conoscenti ti chiedono. Niente feature per "il mercato".

03

Condivisione con conoscenti

3–6 mesi P2 · Medio

Feature che rendono l'app più utile quando la condividi con amici/famiglia: scambio ricette, sync tra tuoi dispositivi, export intelligente. Tutto senza backend — usiamo tecniche P2P o file-based.

Sync tra dispositivi via QR code
Todo
Export del payload corrente come QR code compresso (gzip + base64) sullo schermo. Scan da altro dispositivo → import. Per sync tra tuo phone e tuo desktop senza server. Limitato a ~10KB dati (senza immagini).
P1 ⏱ 3–4 gg → niente backend
Condivisione ricette via link shareable
Todo
Export ricetta singola come JSON compresso in URL fragment (#recipe=...). Apre l'app su un altro dispositivo con modal "Importa questa ricetta?". Per condividere ricette con amici su WhatsApp/Telegram senza backend.
P1 ⏱ 2–3 gg → URL fragment = no server
Backup automatico su Google Drive / iCloud
Todo
OAuth con Google Drive Picker API o iCloud Drive. Backup automatico settimanale del payload come file nutritrack-backup-YYYY-MM-DD.json. Zero costo hosting (gratis entro 15GB utente). Utile per recover in caso di perdita telefono.
P2 ⏱ 5–7 gg → OAuth Google
Multi-profilo per famiglia
Todo
Switch tra profili locali (tu, partner, genitore). Ogni profilo ha il proprio payload separato in localStorage (nutritrack_data_v1_user_alice, ..._bob). Per usare l'app con partner/figli sullo stesso tablet. Niente account.
P2 ⏱ 3–4 gg → estende storage
Web Share API per export
Todo
Usa navigator.share() per export: condividi il backup JSON via WhatsApp/Telegram/email nativamente dal telefono. Più veloce del download + attach manuale.
P3 ⏱ 0.5 gg ✓ niente dipendenze
QR code stampabile per installazione PWA
Todo
Pagina /installa.html con QR code che punta all'URL dell'app. Stampa su biglietto da visita o manda ai tuoi amici via chat. Scan → apre Safari/Chrome → Add to Home Screen. Zero frizione per condividere l'app.
P3 ⏱ 0.5 gg ✓ niente dipendenze
04

Polish e longevità

6–12 mesi P3 · Nice-to-have

Miglioramenti che rendono l'app più robusta, accessibile e mantenibile nel tempo. Niente urgenza, ma rendono il progetto più solido per uso a lungo termine.

Schema migration path
Todo
Implementa migration system per SCHEMA_VERSION (già presente come costante ma non usato). Quando cambi il formato del payload, scrivi migration function migrateV1ToV2() eseguita automaticamente al load. Evita data loss su future evoluzioni.
P2 ⏱ 2–3 gg → estende storage.ts
i18n minimo (IT + EN)
Todo
Estrai stringhe UI in file locales/it.json e locales/en.json. Libreria leggera tipo i18next o custom. IT default, EN per eventuali amici stranieri o contributor. Niente ES/FR per ora (YAGNI).
P2 ⏱ 4–5 gg → refactoring UI
Integrazione widget iOS (manifest + standby)
Todo
Esponi widget "Calorie di oggi" via Web App Manifest (iOS 17+ supporta widget PWA). Richiede display: standalone + shortcuts + eventualmente lock screen widget. Sperimentale su iOS, stabile su Android.
P3 ⏱ 3–4 gg → iOS 17+ only
PWA update notification non-blocking
Todo
Quando il SW rileva un update, mostra toast non invadente "Nuova versione disponibile — ricarica" invece del refresh automatico. Implementa con navigator.serviceWorker.addEventListener('updatefound'). Evita UX jarring durante l'uso.
P3 ⏱ 1–2 gg → estende main.ts
Micro analytics anonimo opt-in
Todo
Opzione in settings: "Aiutami a migliorare l'app". Se attivo, invia a Plausible/Umami self-hosted solo eventi aggregati (app_open, feature_used). Niente PII, niente cookie, GDPR-friendly. Ti dice quali feature usi di più per decidere cosa sviluppare dopo.
P3 ⏱ 2–3 gg → self-host Plausible
03 · Cosa NON fare

Cosa resta fuori dal perimetro

Per mantenere il progetto hobbistico sostenibile, queste feature — pur valide in altri contesti — NON rientrano nella roadmap. Se in futuro cambi idea, lo scenario B o C dell'analisi originale resta disponibile.

🚫

Backend e account

Niente Supabase/Firebase. Sync solo via QR/JSON/file. Se serve multi-device, usa il backup Google Drive della Fase 3.

🚫

App Store / Play Store

Niente Capacitor wrapper, niente review process, niente fee 30%. L'app resta PWA installabile via browser. I tuoi amici la installano da link.

🚫

Monetizzazione

Niente premium tier, niente IAP, niente ads. Se un domani volessi monetizzare, sei libero di forkare — ma per ora resta free OSS puro.

🚫

Wearable nativi (Apple Health/Google Fit)

Richiederebbe Capacitor wrapper → vedi sopra. Per il tracking passivo, usa il diario manuale o integrazione futura via file export da app nativa.

🚫

AI foto cibo

Richiede API a consumo (costo ricorrente) o self-host di modello ML (complesso). Per hobby è overkill. Manual entry resta sufficiente.

🚫

API pubblica per terzi

Richiede auth server, rate limiting, documentazione. Per hobbistica non ha senso — l'API esposta è solo l'app stessa via browser.

Se in futuro cambi idea: l'analisi originale mostra come migrare allo scenario B (Indie premium) o C (OSS community). La base di codice che costruirai nei prossimi 12 mesi resta riusabile — barcode, statistiche, i18n, schema migration sono tutti building block validi per qualsiasi scenario futuro.
